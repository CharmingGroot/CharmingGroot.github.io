# 글쓰기 지침

이 블로그의 글을 쓰거나 다듬을 때 따르는 기준. 볼트(`~/Documents/정리`)에서 원본을 쓸 때 적용한다.

## 말투

- **단정체(~한다 / ~다).** 사실을 단단하게 진술한다. 추측은 추측이라고 밝힌다.
- 1인칭은 판단·회고가 필요할 때만 절제해서 쓴다. "써보니 배치 실패가 전체를 막았다", "이 설계는 과하다고 본다" 정도. 남발하지 않는다.
- 독자에게 가르치려 들거나 친한 척하지 않는다. 동료에게 정리해서 건네는 톤.

## 구성

한 편은 세 덩어리로 간다.

1. **도입 훅 (1~2문단).** 왜 이걸 보는지, 무슨 문제를 푸는지 먼저. 큰 그림을 깔고 들어간다. 깊은 디테일은 그다음.
2. **본문.** H2/H3로 구조화. 표·코드블록·다이어그램을 적극 쓴다. 깊이는 내부 동작·아키텍처까지 파고든다.
3. **정리.** 마지막에 핵심을 한 문단으로 압축하고, 한계나 다음에 볼 것을 짧게 남긴다.

제목은 "주제 — 한 줄 설명" 또는 시리즈 번호(`보안-NN. …`) 형식.

## 깊이

- **깊게 간다.** 개념만 훑지 않고 내부 동작·구조·트레이드오프까지. 이게 차별점이다.
- 단, 깊은 디테일 앞에 항상 큰 그림(왜/무엇)을 먼저 준다. 맥락 없이 디테일부터 던지지 않는다.
- 코드·설정은 **실제로 동작하는 것**만 싣는다. 지어낸 예제 금지.
- 외부 출처를 가져왔으면 frontmatter `source` 또는 본문에 링크로 밝힌다.

## 덜어낼 것 (AI 티)

- 칭찬·인사성 오프닝 ("좋은 질문이다", "흥미롭게도")
- 마케팅·거창한 프레이밍 ("seamless", "강력한", "혁신적인")
- 템플릿식 헤더 떡칠 — 문단으로 흘려도 되는 걸 굳이 소제목으로 쪼개기
- em-dash(—) 남발, 불릿 과다
- 다 적으려는 포괄 강박 — 핵심부터, 곁가지는 쳐낸다

```yaml
title: "보안-07. 제목 — 한 줄 설명"
date: 2026-06-20
tags: [security, falco, ...]   # 첫 태그는 큰 분류, 나머지는 구체 키워드
summary: "이 글이 무엇을 다루는지 2~3문장. 목록 첫 화면에 노출된다."
# source: "https://..."        # 번역/인용이면
---
```
